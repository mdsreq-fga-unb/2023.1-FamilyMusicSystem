    O backlog do produto é uma lista ordenada que contém todos os requisitos que possam ser necessários para desenvolvimento de um projeto, podendo conter: epics, features e users stories. Além disso, é a única fonte de requisitos para todas as mudanças a serem feitas no produto. O time recorrerá a ele durante o processo para eventuais dúvidas sobre projeto.

    Personas

    • Professor: É o usuário comum que é capaz de ter o controle de turmas e alunos.
    • Aluno: É o usuário comum que é capaz de ter acesso aos boletos de sua mensalidade e possíveis avisos provenientes do administrador e professor.
    • Administrador: Pessoa com funções que vão desde o gerenciamento da escola de música até o envio de mensagens de aviso para os alunos.
    • Usuário: Refere-se tanto ao aluno quanto ao professor.

    Requisitos funcionais

    Os requisitos funcionais são as funcionalidades que o software deve realizar, e dependem de diversos fatores, como a abordagem utilizada para descrever os requisitos, o usuário do software, dentre outros. No projeto FMS, a análise de viabilidade do projeto conta com três critérios: Valor de negócio (N), Viabilidade (V) e Criticidade (C). Na tabela a seguir, cada requisito recebe uma nota de 1 a 5 atribuindo valor para esse requisito, e cada um receberá seu grau de prioridade de acordo com a média desses critérios para cada requisito: (N+V+C)/3.

    [TE01] [EP01] [US06] - Deve haver uma confirmação para o administrador conferir se realmente deseja excluir o perfil do aluno
    - O administrador removerá o aluno, mas suas aulas e presenças continuarão presentes na base de dados;
    [TE01] [EP02] [US07] - O usuário deverá ter um CPF e senha registrados para conseguir acessar o sistema.
    - O usuário deverá receber uma confirmação de acesso, caso suas credenciais estejam corretas.
    [TE01] [EP02] [US08] - O usuário deverá fornecer seu e-mail, CPF e data de nascimento para que uma nova senha seja enviada via correio eletrônico;
    - O usuário deverá trocar a senha temporária após entrar na aplicação;
    - O usuário terá o tempo mínimo de 30 minutos entre uma recuperação de senha e outra;
    [TE02] [EP03] [US09] - O administrador deverá informar as características da sala: nome da sala, descrição da sala e capacidade de alunos;
    - O administrador não poderá hospedar uma aula numa sala que não foi criada;
    - Não poderão haver salas com o mesmo número;
    [TE02] [EP03] [US10] - Não será possível editar uma sala não cadastrada;
    - Não poderá adicionar duas aulas para a mesma sala;
    - Não poderá editar o nome da sala para um nome que já está cadastrado;
    [TE02] [EP04] [US11] - Não será possível registrar uma aula em um horário não disponível para o professor;
    - Não será possível registrar uma aula em uma sala não existente;
    - Não será possível cadastrar uma aula em uma sala em que já ocorre uma aula nesse mesmo horário;
    [TE02] [EP04] [US12] -Para agendar, devem ser solicitados: nome do professor, horário, tipo de aula (coletiva ou individual) que deverá estar disposta em um campo de seleção e o instrumento;
    -Se a aula for coletiva , deverá ser solicitado o número da turma, e se for individual, o nome do aluno;
    - O horário escolhido deverá estar de acordo com o horário de funcionamento das aulas;
    - O administrador deve realizar um agendamento de aula sem que o software permita horários que se choquem;
    - As aulas agendadas devem ser registradas no banco de dados;
    - O administrador deve ver a confirmação do agendamento;
    [TE02] [EP04] [US13] - O administrador deve conseguir filtrar as aulas agendadas por dia, mês e ano;
    - Deve ser possível visualizar o nome do professor que dará a aula, o tipo da aula (coletiva ou individual), se for individual mostrar o nome do aluno e se for coletiva mostrar o número da turma, o instrumento, o dia e horário;
    - Deve ser visível uma área para edição da aula agendada;
    - Deve ser visível uma área para remoção da aula agendada;
    [TE02] [EP04] [US14] - O professor deve conseguir editar aulas anteriormente marcadas;
    - Devem ser solicitados os dados de alteração: tipo de aula (coletiva ou individual)(número da turma ou nome do aluno), instrumento, dia e horário;
    - Deve ser perguntado se o professor realmente deseja fazer essa alteração;
    - O professor deve visualizar as alteraçōes realizadas;
    - As alterações devem ser salvas no banco de dados;

    Requisitos não-funcionais

    Categoria Requisito
    Implementação [RNF01] O back-end da aplicação deve ser construído com a ferramenta Strapi.
    Interface [RNF02] A interface deve conter modo de alto contraste e texto alternativo, com o propósito de deixar a aplicação mais acessível.
    Interface [RNF03] A aplicação deve conter estilização das páginas com cores, fontes e icones padronizados.
    Usabilidade [RNF04] A aplicação deve ser responsiva, podendo ser acessada por diferentes tamanhos de telas.
    Usabilidade [RNF05] A aplicação deve possuir um manual de ajuda para instruir novos usuários na plataforma.
    Usabilidade [RNF06] A aplicação deve possuir uma aba intitulada "Sobre" contendo uma breve explicação a respeito da aplicação.
    Suportabilidade [RNF07] A aplicação deve ser compatível com as versões mais recentes dos navegadores modernos como Chrome, Safari, Edge, Firefox, e Opera.
    Confiabilidade [RNF08] A aplicação deve garantir a privacidade dos dados dos usuários, estando de acordo com a Lei Geral de Proteção de Dados (13.709/2018).
    Confiabilidade [RNF09] A aplicação deve proporcionar um ambiente suficientemente seguro para garantir que apenas aqueles registrados no sistema tenham acesso a ele.
    Confiabilidade [RNF10] A aplicação deve possuir Política de Privacidade clara e transparente para com o cliente
